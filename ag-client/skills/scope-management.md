# Scope Management
**Purpose:** Manage scope so projects stay profitable and clients stay happy — because scope creep is the single biggest threat to agency profitability, and the solution is not saying no, it's saying yes correctly.

Scope creep doesn't happen in big, obvious moments. It happens in small, reasonable requests: "Can you also adjust the mobile version?" "Could you add one more page?" "The CEO wants to see a different direction." Each request is small. Collectively, they can double the project hours without a single amendment to the SOW.

## The Scope Definition
The SOW should define deliverables by output, not by effort. "One responsive landing page with hero, features, pricing, testimonials, and CTA sections" is a defined scope. "Landing page design and development" is an invitation for scope creep. Every deliverable should be specific enough that both parties can independently determine whether it's been delivered.

## The Change Request Protocol
When a client requests something outside scope: acknowledge the request positively ("great idea"). Assess the impact honestly ("this would add approximately 2 days to the timeline and $X to the budget"). Present options ("we can add this to the current scope with a change order, or we can note it for a follow-up phase"). Get written approval before starting. Never absorb scope additions silently — it trains the client to expect free additions.

## The Prevention Strategy
The best scope management happens before the project starts. Include explicit exclusions in the SOW ("this engagement does not include: copywriting, photography, backend development, ongoing maintenance"). Define revision rounds ("includes 2 rounds of design revisions"). Specify what constitutes a revision vs a new direction.

## When to use
Every project. When projects consistently go over budget. When the team is working more hours than scoped. When client expectations don't match the SOW.
